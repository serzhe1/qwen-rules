
# Глобальные правила работы агента

## Общий стиль работы
- Всегда ориентируйся на стиль кода текущего репозитория.
- Перед изменением кода изучи соседние файлы и существующие паттерны.
- Не создавай новые классы, пакеты, абстракции или утилиты, если рядом уже есть аналогичное решение.
- Предпочитай минимальные изменения вместо широкого рефакторинга.

## Планирование (plan mode)
- В режиме планирования сначала задай уточняющие вопросы.
- Не придумывай требования, если они не указаны явно.
- Если план затрагивает более 3 файлов, сначала покажи план изменений и дождись подтверждения пользователя.

## Приоритет локального стиля
- Локальные соглашения репозитория важнее общих best practices.
- Если в проекте уже есть пример реализации — копируй этот подход.
- Если примеры противоречат друг другу, сначала задай вопрос.

## После ручных исправлений пользователя
- Если пользователь вручную изменил код, считай текущую версию файла источником истины.
- Не возвращай предыдущую версию кода, которую предложил агент.
- Продолжай работу от текущего состояния файлов.

## Проверка сборки
После внесения изменений:
- попробуй собрать проект
- убедись, что проект хотя бы компилируется

Используй команды:

./gradlew build

или

./gradlew test

## Тесты

Если тесты уже есть:
- копируй стиль существующих тестов
- следуй их структуре и используемым библиотекам

Если тестов нет:

### Unit тесты
- JUnit 5
- Mockito
- AssertJ

### Интеграционные тесты
- SpringBootTest

### E2E тесты
- Testcontainers через библиотеку команды `testing-sb3`

### REST API тесты
- RestAssured

### Моки внешних сервисов
- WireMock

## Ограничения на запуск тестов
- Не запускай тяжелые интеграционные или e2e тесты автоматически.
- Запускай только минимальную проверку, необходимую для компиляции и базовой валидации изменений.
